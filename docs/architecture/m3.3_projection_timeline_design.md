## 一、Document Information

### 1.1 Purpose

本文定义 **AssetSnapshot Projection**、**AssetTimeline Projection** 及未来扩展投影的 **Projection Contract**、Registry、Builder 生命周期、依赖图、健康状态与统一 DTO，以及其与 M3.2 Workbench Application Service 的衔接方式。

## 四、Projection Principles

### 4.1 Projection over Synchronization（Principle 7）

写事实 COMMIT 后，投影 **失效 → 异步构建 → Published**；不以同步双写为权威。

### 4.2 Domain First（Principle 10）

**Projection 是唯一对外 Read Model**（ADR-PRJ-009）。事实表仅 Builder `collect()` 可读。

### 4.3 可重建性（Principle 5）

给定 `identity_id` + 事实快照，确定性 `rebuild()` 产出相同 DTO（同 `builder_version`）。

### 4.4 追加式历史（Principle 9）

Timeline 只追加；`correlation_id` 用于 UI 聚合，不替代事件行。

### 4.5 身份边界

所有 Builder 输入必须先有 `identity_id`（M3.1 解析）。

## 五、Projection Contract

**以后任何 Projection（含未来 Cashflow / Portfolio）都必须符合本 Contract。** M3.4 只引用 Contract 产出的 DTO，不另定义字段语义。

### 5.1 全局不变量（AssetSnapshot & 一切 Snapshot 类投影）

每个 Snapshot 类投影 **必须** 满足：

| 属性 | 含义 |
|------|------|
| **Consistency** | 同一 `snapshot_key` + 同一事实水位下，多次 `publish()` 产出业务字段一致 |
| **Idempotent** | 重复 `publish()` / Rebuild 不重复副作用、不 duplicate 展示行 |
| **Rebuildable** | 仅只读事实即可全量重建；不依赖不可重放的隐藏状态 |
| **Observable** | 必须暴露 `built_at`、`projection_version`、`builder_version`、Health（§十八） |
| **Eventually Fresh** | 事实变更后终将在 SLA 内变为 `Fresh`；允许短暂 `Stale` |

Timeline 类投影同样满足 **Idempotent、Rebuildable、Observable、Eventually Fresh**；Consistency 指排序与 `event_id` 稳定。

### 5.2 每个 Projection 必须声明的注册项

在 **ProjectionRegistry**（§七）中，每个投影 **必须** 登记：

| 声明项 | 说明 |
|--------|------|
| **Builder** | 实现统一生命周期的 Builder 类（§八） |
| **Refresh Trigger** | 哪些事实/域事件触发 `invalidate()` |
| **Cache Policy** | L0/L1/L2、TTL、`snapshot_key` 规则 |
| **Rebuild Strategy** | 单 Identity / 批量 / 全库；是否阻塞读 |
| **Consumer** | 谁读此投影：`workbench` / `api` / `bi` / `ai` / `internal` |

未在 Registry 登记的投影 **不得** 被 Workbench / API / BI / AI 使用。

### 5.3 Contract 与 M3.4 的关系

M3.4 Open API 响应 schema **= Registry 中投影的 Unified DTO + 元数据信封**（§六、§十）。禁止在 M3.4 文档中重新定义 Snapshot 字段含义。

## 六、Unified Projection DTO

**禁止** `WorkbenchDTO`、`AgentDTO`、`BIDTO` 各搞一套。Web、API、BI、AI **全部复用**下列逻辑对象（M2 `object_dictionary` 同步登记）。

### 6.1 元数据信封（所有投影响应必带）

| 字段 | 说明 |
|------|------|
| `identity_id` | Aggregate Root |
| `projection_kind` | `asset_snapshot` \| `asset_timeline` \| `asset_operations` \| … |
| `built_at` | 本次构建完成时间（ISO 8601） |
| `projection_version` | 投影存储逻辑版本（如 `m3.3-v1`） |
| `builder_version` | Builder 代码/规则版本（Rebuild 归因用） |
| `snapshot_version` | 单键单调递增（per `snapshot_key`）；每次 `publish()` +1 |
| `health` | `fresh` \| `stale` \| `building` \| `failed` \| `rebuilding`（§十八） |
| `source_watermark` | 事实源水位摘要（调试用） |

### 6.2 核心 DTO

| DTO | 用途 | 对应 M3.2 |
|-----|------|-----------|
| **`AssetSnapshotDTO`** | 单扇区或聚合快照 payload + 元数据信封 | `fetch_snapshot` 各 `*Snapshot` |
| **`AssetTimelineDTO`** | `events[]` + 分页 + 元数据信封 | `fetch_timeline` → `TimelinePage` |
| **`AssetOperationsDTO`** | 运营任务视图 + 元数据信封 | `fetch_operations` |
| **`AssetIdentityDTO`** | Identity 元数据（非投影，但响应可带 `projection_health` 摘要） | `fetch_asset_identity` |

### 6.3 扇区 Payload（`AssetSnapshotDTO.payload`）

扇区类型 `sector` 与 M3.2 §5.2 一致：`issuance` | `monitor` | `repayment` | `risk` | `trust` | `operations`。

未来扩展（`cashflow`、`portfolio`）须新 Register + 新 `sector` 枚举，**升 M3.3 Minor**。

### 6.4 禁止平行 DTO

| 禁止 | 采用 |
|------|------|
| API 专用 Snapshot JSON | `AssetSnapshotDTO` |
| AI Tool 扁平化 ad-hoc 结构 | `AssetSnapshotDTO` + 可选 `fetch_ai_summary` 摘要层 |
| BI 导出另建宽表 schema | `AssetSnapshotDTO` 批量接口（M3.4） |

## 七、Projection Registry

### 7.1 定位

**ProjectionRegistry** = 全平台投影 Builder 的**唯一目录**。Workbench Application Service **只**通过 Registry 解析 `projection_kind` → Builder。

```
fetch_snapshot(sector)  →  Registry.get("asset_snapshot", sector)  →  Builder
fetch_timeline()        →  Registry.get("asset_timeline")           →  AssetTimelineBuilder
```

### 7.2 初版登记（V1.1）

| projection_kind | Builder | Consumer | 说明 |
|-----------------|---------|----------|------|
| `asset_snapshot` / `issuance` | `IssuanceSnapshotBuilder` | workbench, api, bi, ai | SSOT 发行 |
| `asset_snapshot` / `monitor` | `MonitorSnapshotBuilder` | workbench, api, bi, ai | as-of 监控 |
| `asset_snapshot` / `repayment` | `RepaymentSnapshotBuilder` | workbench, api, bi, ai | 累计 + 明细分页 |
| `asset_snapshot` / `risk` | `RiskSnapshotBuilder` | workbench, api, bi, ai | 风险摘要 |
| `asset_snapshot` / `trust` | `TrustSnapshotBuilder` | workbench, api, bi | 产品上下文 |
| `asset_snapshot` / `operations` | `OperationsSnapshotBuilder` | workbench, api, ai | 跟进摘要 |
| `asset_timeline` | `AssetTimelineBuilder` | workbench, api, bi, ai | 统一事件流 |

### 7.3 未来扩展（须 Register + ADR）

| Builder（预留名） | 说明 |
|-------------------|------|
| `RiskSnapshotBuilder` | 若独立于扇区 `risk` 的跨资产风险视图 |

## 八、Builder Lifecycle

**所有 Builder 必须实现统一生命周期**；禁止各模块私自实现长短不一的构建流程。

### 8.1 标准阶段

```
prepare() → collect() → aggregate() → validate() → publish() → [invalidate()]
```

| 阶段 | 职责 | 可读层 |
|------|------|--------|
| **`prepare()`** | 解析 `identity_id`、IdentityScope、参数校验；状态 → `Building` | Identity（M3.1） |
| **`collect()`** | 只读拉取本 Builder 依赖的事实行 | **仅**事实表（内部） |
| **`aggregate()`** | 业务聚合、as-of 切片、扇区内计算 | 内存 |
| **`validate()`** | Contract 校验、跨扇区核对钩子；失败 → `Failed` | — |
| **`publish()`** | 写入缓存/L2、递增 `snapshot_version`、状态 → `Published` / `Fresh` | 投影存储 |
| **`invalidate()`** | 标记 `Invalid`、bump `inv:{identity_id}`；不删事实 | 投影元数据 |

Timeline Builder 无 `sector` 时：`collect()` 多源事实 + Repair/Alias 日志；`aggregate()` 归并排序；`publish()` 物化事件集。

### 8.2 与 State Machine 的关系

各阶段只应在 §九 允许的状态转移下调用（如 `Invalid` 后才能 `prepare()` 重建）。

### 8.3 测试要求

- 单元测试：可 mock `collect()` 输入，断言 `publish()` DTO
- 集成测试：经 M3.2 Service，**不**直接测 HTTP

## 九、Projection State Machine

每个 `snapshot_key`（或 Timeline 的 `identity_id` 键）在任意时刻处于下列状态之一：

```
                    ┌─────────┐
                    │  Draft  │  （可选：预计算未对外）
                    └────┬────┘
                         │ prepare()
                         ▼
                    ┌─────────┐
         ┌─────────│ Building│─────────┐
         │         └────┬────┘         │
         │ validate fail│ publish()   │ invalidate()
         ▼              ▼             ▼
    ┌─────────┐   ┌───────────┐   ┌─────────┐
    │ Failed  │   │ Published │   │ Invalid │
    └─────────┘   └─────┬─────┘   └────┬────┘
                        │              │
              SLA 过期 / 事实更新      │ rebuild
                        ▼              ▼
                   ┌─────────┐   ┌────────────┐
                   │  Stale  │   │ Rebuilding │
                   └─────────┘   └──────┬─────┘
                        │               │ publish()
                        └───────┬───────┘
                                ▼
                          Published / Fresh
```

| 状态 | 含义 | 对外读 |
|------|------|--------|
| **Draft** | 构建中未发布 | 不返回 |
| **Building** | `prepare`～`validate` | 可返回旧 `Published` + `health=building` |
| **Published** | 可用 | 返回 DTO |
| **Invalid** | 已失效待重建 | 返回旧 DTO + `health=stale` 或触发同步 rebuild |
| **Stale** | 已发布但水位过期 | 返回 DTO + `health=stale` |
| **Rebuilding** | 全量 rebuild 中 | 同 Building |
| **Failed** | `validate` 或异常 | `health=failed`；AI **不得**基于该扇区作答（§十八） |
| **Archived** | Identity 归档/下线 | 只读历史；不刷新 |

**禁止**仅有「Refresh」而无状态：一切 Builder 走本状态机。

## 十、AssetSnapshot Projection

### 10.1 逻辑对象

**AssetSnapshotProjection** = 单一 `identity_id` + `sector` + 参数 → **`AssetSnapshotDTO`**。

| 逻辑键 | 组成 |
|--------|------|
| `snapshot_key` | `identity_id` + `sector` + `param_hash` |
| `snapshot_version` | per-key 单调递增（每次 `publish()`） |
| `built_at` | 构建完成时刻 |
| `projection_version` | 投影模式版本 |
| `builder_version` | Builder 规则版本 |
| `payload` | 扇区业务字段 |
| `source_watermark` | 事实水位 |

### 10.2 Workbench Header 投影元数据（M3.2 Header 扩展）

Header **必须**展示默认 Monitor 扇区（或聚合）的投影健康度，供人与 AI 判断可信度：

| Header 字段 | 示例 |
|-------------|------|
| `identity_id` | `42` |
| `snapshot_health` | `fresh` |
| `snapshot_built_at` | `2026-06-23T10:15:00Z` |
| `snapshot_age_human` | `3 mins ago` |
| `projection_version` | `m3.3-v1` |
| `builder_version` | `issuance-1.2.0` |

展示语义（非视觉稿）：`Snapshot · Fresh · 3 mins ago`。

### 10.3 各扇区规则（继承 M3.2 §5.2）

| Sector | 业务 as-of | Builder |
|--------|------------|---------|
| **issuance** | `issue_date` | `IssuanceSnapshotBuilder` |
| **monitor** | `data_date` | `MonitorSnapshotBuilder` |
| **repayment** | 累计 + 区间 | `RepaymentSnapshotBuilder` |
| **risk** | monitor as-of | `RiskSnapshotBuilder` |
| **trust** | 产品主数据 | `TrustSnapshotBuilder` |
| **operations** | 当前运营态 | `OperationsSnapshotBuilder` |

### 10.4 Identity 范围与跨扇区一致性

同 V1.0 §5.3–5.4：`IdentityScopeResolver` 在 `prepare()`；差异可产生 `RECONCILIATION` 事件。

## 十二、Event Model

### 12.1 AssetEvent（继承 M3.2 §6.2）

| 字段 | 说明 |
|------|------|
| `event_id` | 全局唯一 |
| `identity_id` | 归属 Root |
| `event_type` | 见 §12.2 |
| `occurred_at` / `recorded_at` | 业务 / 系统时间 |
| `source_module` / `source_ref` | 溯源 |
| **`correlation_id`** | **强烈建议必填**（§12.3） |
| `correlation_kind` | `import_batch` \| `repair_job` \| `risk_recalc` \| `manual_action` \| `alias_change` |
| `payload_summary` / `payload` | 展示与扩展 |
| `actor` / `visibility` | 审计 |

### 12.2 event_type 枚举

同 V1.0 §7.2（`ISSUANCE` … `LINEAGE_LINKED`）。

### 12.3 Event Correlation（强烈建议）

**问题：** 单次 Import / Repair / 风险重算往往产生多条 `MONITOR_IMPORTED`、`REPAIR`、`RISK_*` 事件，Timeline 难读。

**规则：**

| 场景 | `correlation_id` | `correlation_kind` |
|------|------------------|---------------------|
| Import Batch | `import_batch_id` | `import_batch` |
| Repair Job | `repair_log.id` 或 `repair_name+run_id` | `repair_job` |
| Risk 重算 | `risk_recalc_run_id` | `risk_recalc` |
| 人工动作 | `manual_action_id` | `manual_action` |
| Alias 批量 | `alias_change_batch_id` | `alias_change` |

**Workbench Timeline UI（信息架构，非视觉）：**

- 默认：按时间平铺
- 可选：`group_by=correlation_id` 折叠为「一次操作 · N 条事件」
- API（M3.4）：`fetch_timeline?group_by=correlation_id`

同一 `correlation_id` 下事件 **共享** `occurred_at` 窗口与 `payload.batch_summary`（可选）。

### 12.4 事件与事实

事件描述变更，不替代事实；Repair 必须引用 `repair_log` 逻辑 id。

## 十三、Event Source Mapping

## 十四、Projection Dependency Graph

Refresh / Invalidate **必须**按依赖图自动推导，禁止手写散落 if-else。

### 14.1 图（逻辑）

```
                    Issuance
                        │
                        ▼
    Monitor ──────────┬──────────────┐
        │           │              │
        ▼           ▼              ▼
    Repayment ──► AssetSnapshot ◄── Risk
        │           ▲              │
        │           │              │
        └───────────┼──────────────┘
                    │
              Followup / Operations
                    │
                    ▼
              AssetTimeline
         （Issuance / Import / Repair /
          Alias / Followup 等皆汇入）
```

### 14.2 依赖边（失效传播）

| 上游事实变更 | 直接失效 | 级联失效 |
|--------------|----------|----------|
| Issuance | `issuance` | `monitor`, `repayment`, `risk` |
| Monitor | `monitor` | `risk` |
| Repayment | `repayment` | —（可选 reconcile → `monitor`） |
| Risk | `risk` | — |
| Followup | `operations` | Timeline 追加（不反向失效 Snapshot） |
| Alias / Identity | **全部** Snapshot 扇区 | Timeline 追加 |
| Repair | 按 repair 影响表查表 | 依赖图自动扩展 |

### 14.3 与 Refresh 的衔接

```
on_fact_change(event):
    affected = dependency_graph.resolve(event.fact_kind)
    for each node in affected:
        registry.get(node).invalidate(identity_id)
    if event.emits_timeline:
        asset_timeline_builder.append(event)
```

### 14.4 Timeline 与 Snapshot 关系

- Snapshot 扇区 **不** 反向依赖 Timeline 数据构建
- Timeline **可** 引用 Snapshot `validate()` 产生的 `RECONCILIATION` 事件
- Rebuild 顺序建议：**Timeline → Snapshot 叶节点 → 根依赖扇区**（或并行 Snapshot + 最后 reconcile）

### 15.1 操作定义

| 操作 | 说明 |
|------|------|

| **Refresh** | 单键 `prepare→publish` |

|------|----------|-------------------|

| Alias | `ALIAS_CHANGED` | 全部 Snapshot |
| Repair | `REPAIR` | `repair_scope` 解析 → 图遍历 |
| Followup | `FOLLOWUP_*` | `operations` |

### 15.3 SLA 与事务边界

同 V1.0：事实 COMMIT 后事务外 `invalidate`；**Eventually Fresh**。

## 十六、Projection Rebuild Strategy

### 16.1 何时 Rebuild

投影 bug、`builder_version` 变更、Repair 大规模、管理员 `rebuild_projection(identity_id)`。

### 16.2 流程

```
1. 状态 → Rebuilding
2. AssetTimelineBuilder.rebuild()（全量重放 Source Mapping）
3. 按依赖图逆拓扑或 Registry 顺序 rebuild 各 Snapshot Builder
4. 更新 projection_version / builder_version 戳
5. 状态 → Published / Fresh；写 rebuild_audit
```

只读事实；不锁事实表。

## 十八、Projection Health

### 18.1 健康枚举

| health | 含义 | Header 展示 | AI 行为 |
|--------|------|-------------|---------|
| **fresh** | 水位内且 `Published` | `Fresh · N mins ago` | 正常作答 |
| **stale** | 已失效或 SLA 过期 | `Stale · rebuilding…` | 答案须注明数据可能滞后 |
| **building** | 首次构建中 | `Building…` | 等待或读旧快照并声明 |
| **rebuilding** | 全量重建中 | `Rebuilding…` | 同 building |
| **failed** | `validate` 失败 | `Failed` | **禁止**对该扇区作确定性陈述 |

### 18.2 聚合 Header 规则

- 默认展示 **Monitor** 扇区 health（与 as-of 一致）
- 任一扇区 `failed` → Header 级警告（信息架构）
- `fetch_ai_summary` **必须**传入 `health`；M3.5 据此约束幻觉

### 18.3 观测

- Metrics：`projection_build_duration`, `stale_count`, `failed_count`
- 日志：`identity_id`, `snapshot_key`, `builder_version`, `correlation_id`

### ADR-PRJ-009：Projection is the Single Read Model（**Domain 最重要 ADR 之一**）

**结论：** Workbench、Open API、BI、AI **不得**自行 Join 事实表；**必须**读取 **Projection DTO**（经 M3.2 Application Service）。

**理由：**

1. **Principle 10 Domain First**：Presentation 面向 Domain，非表结构。
2. **单一读模型**：避免 Web/API/BI/AI 四套 Join 逻辑漂移（Product3 类一致性问题在读侧重演）。
3. **可观测与 Debug**：`built_at`、`builder_version`、`health` 只在 Projection 信封中统一。
4. **M3.4 简化**：API 即 DTO 序列化，不重复定义聚合逻辑。

**例外（须登记 ADR 并限期收敛）：**

- 产品级 / 组合级 **非单资产** 报表可注册 `ProductSnapshotBuilder` / `PortfolioSnapshotBuilder`，仍走 Registry，**不算**直连事实表的 ad-hoc Join。

### ADR-PRJ-010：为什么强制 Projection Contract 五元组登记？

无 Registry 声明则 Refresh/Rebuild/Cache 无法依赖图自动化；新投影会复制粘贴失效逻辑。

### ADR-PRJ-011：为什么 Builder 统一六阶段生命周期？

避免 Issuance 与 Timeline 各自实现「半套缓存半套查询」；State Machine 可统一观测。

### ADR-PRJ-012：为什么 correlation_id 强烈建议必填？

Import/Repair 是多表、多事件一次操作；无 correlation 则 Timeline 不可读、AI 无法归纳「上一次导入发生了什么」。

## 二十八、Freeze Statement

**M3.3《Projection & Timeline Design》V1.1** 已评审通过，自发布之日起**冻结**，作为整个 Asset Domain **Projection Layer 的唯一架构依据（Projection Layer Baseline）**。

### 28.1 唯一依据范围

- Projection Contract、ProjectionRegistry、Builder 生命周期与 State Machine
- Unified Projection DTO（`AssetSnapshotDTO`、`AssetTimelineDTO`、`AssetOperationsDTO`）
- Event Model、Event Source Mapping、`correlation_id` 规则
- Refresh / Rebuild / Cache / Health / Projection Dependency Graph
- M3.2 `fetch_snapshot` / `fetch_timeline` 的内部读路径实现

### 28.2 强制读路径

**Workbench、Open API、BI、AI** 均必须：

1. 经 **M3.2 Application Service**（`fetch_asset_identity`、`fetch_snapshot`、`fetch_timeline` 等）访问资产读模型；
2. 读取 **M3.3 Unified Projection DTO**（含 `built_at`、`projection_version`、`builder_version`、`health` 元数据信封）。

**禁止：**

- 绕过 Projection **直接 Join 或查询事实表**作为 Read Model（**ADR-PRJ-009**）；
- 绕过 **ProjectionRegistry** 私自挂载未登记 Builder；
- 新增 `WorkbenchDTO`、`AgentDTO`、`BIDTO` 等**平行 DTO**。

### 28.3 下游文档边界（M3.4 / M3.5）

**M3.4 Domain API** 与 **M3.5 AI Asset Agent** **不得**重新定义：

- Projection Contract（§五）
- Builder Lifecycle（§八）
- Unified Projection DTO 字段语义（§六）
- Timeline Event Model（§十二）

M3.4 仅将 M3.2 Service 映射为 REST，响应体 **= M3.3 DTO**（§二十四）。M3.5 Tool 响应 **= M3.4 同 schema** 或 `fetch_ai_summary` 文本层。

### 28.4 身份与演进

**不得**在投影实现中重新定义 `AssetIdentity`、Alias、`identity_id`、Canonical Identity；身份问题 **只引用 M3.0 / M3.1**。

**M3.3 冻结后：** 缓存介质、调度实现、物化存储可选型可迭代；**Contract、DTO 形状、Registry 登记项、依赖图、Event Model、M3.4/M3.5 读边界** 变更须升 **M3.3 V1.2+** 或回溯 M3.0 / M3.2 Baseline。
