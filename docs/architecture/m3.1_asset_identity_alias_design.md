## 三、AssetIdentity Design

### 3.1 Aggregate Root（Principle 11）

```
AssetIdentity（唯一 Aggregate Root）
 ├── AssetAlias
 ├── AssetSnapshot（M3.3 投影，M3.1 仅预留挂载点）
 ├── AssetTimeline（M3.3 投影，M3.1 仅预留挂载点）
 └── AssetWorkbench（M3.2，依附 Identity 聚合）
```

Alias、Snapshot、Timeline、Workbench **均不得脱离** `AssetIdentity` 独立存在。

### 3.2 Canonical Identity

**保持 M3.0 §六 定义，不得修改：**

```text
trust_product_id + issue_date + canonical_custody_asset_code
```

`canonical_custody_asset_code` = **发行时托管房源编号**（Anchor Identity）。

### 3.3 identity_id

| 项 | 说明 |
|----|------|
| **定位** | 内部代理键（Surrogate Key） |
| **职责** | 内部 Join；Alias 挂载；Snapshot / Timeline 聚合（M3.2+） |
| **不得** | 作为长期开放 API 的唯一对外标识 |
| **对外** | 长期保持 Canonical **三元组**（M3.0 §十三） |

**建议物理表（Migration 设计，本阶段不执行）：** `asset_identities`

| 字段 | 说明 |
|------|------|
| `id` | 即 `identity_id` |
| `trust_product_id` | |
| `issue_date` | |
| `canonical_custody_asset_code` | |
| `ssot_issuance_record_id` | 权威发行行 |
| `primary_trust_asset_id` | Persistence 默认 FK |
| `created_at` / `updated_at` | |

```text
UNIQUE (trust_product_id, issue_date, canonical_custody_asset_code)
```

> `business_asset_key` 与上述公式相同，但指标的是**发行表可多行**；Identity 表为 **Canonical 级 1:1**。不得将 `business_asset_key` 作主键或 `identity_id` 生成依据（ADR-006）。

### 3.4 SSOT Rule

发行表 `trust_product_issuance_asset_records` 允许同 Canonical 多行。每个 `AssetIdentity` **仅一条**。

**SSOT 选择顺序：**

1. `migration_type = 'new_issuance'` 且 custody 匹配 → **最早** `id`
2. 否则同键 → **最早** `id`
3. 仍无法唯一确定 → **`needs_confirm`**，禁止自动覆盖

人工指定 `ssot_issuance_record_id` 须走 Production Repair 审计。

## 四、trust_asset_aliases

### 4.1 Purpose

保存 **运营编号 → Canonical Identity** 映射（M3.0 §七）。

与 **`trust_product_aliases`**（产品名称）严格区分。

### 4.2 Alias Type（初版）

| 值 | 含义 |
|----|------|
| `issuance` | 发行种子（= Canonical 自身） |
| `external_change` | 合作机构调整后的运营编号 |
| `manual` | 人工确认映射 |
| `repair` | Production Repair 写入 |
| `legacy` | Backfill 推断候选经确认后 |

预留扩展；新增类型须更新 Canonical `alias_dictionary`。

### 4.3 Constraints（§七）

| 允许 | 禁止 |
|------|------|
| 多个 Alias → 一个 Canonical | 一个有效 Alias → 多个 Canonical |

**建议约束：**

```text
FK: identity_id → asset_identities.id
UNIQUE (trust_product_id, alias_custody_asset_code) WHERE expired_date IS NULL
```

每个 Identity 创建时自动插入 `alias_type=issuance` 种子行。

### 4.4 Lifecycle

```
Created → Active → Expired → Archived
```

- 历史 Alias **永久保留**（Principle 9：Timeline 追加，不覆盖）
- 失效通过 `expired_date` 标记，不物理删除

## 五、Alias Resolution

统一解析器：`resolve_alias(trust_product_id, imported_custody, as_of_date?)`

**流程（对齐 M3.0 §七）：**

```
Step 0  规范化（clean_custody_code，去 .0）

Step 1  Exact — 命中 asset_identities.canonical_custody_asset_code

Step 2  Active Alias — expired_date IS NULL，唯一 identity_id

Step 3  Historical Alias — effective_date..expired_date 窗口内匹配

Step 4  Issuance Readonly Mapping — 只读对照 issuance 表，用于 Backfill 提示
        不作为静默导入路径

Step 5  Failed — unresolved
```

| 结果 | 行为 |
|------|------|
| 唯一命中 | `resolved` |
| 多 Canonical 候选 | `ambiguous` → **`needs_confirm`** |
| 无命中 | `unresolved` → 默认 **`needs_confirm`**；严格模式 `reject` |

**Principle 4 / ADR-005：Workbench Never Guess。**

`source_asset_code` **不参与** Alias 键解析（Principle 8）；分笔归属在 Operation 层单独处理。

## 六、Import Resolution（§十四）

```
Excel 行
  ↓
[已有] 权威列 / 双列不一致 → needs_confirm
  ↓
resolve_alias → Canonical + identity_id
  ↓
Business Validation（行数、金额、重复等）
  ↓
Import（须 confirm_sheet_keys 若 needs_confirm）
```

**写库保留三层（§十四）：**

| 层 | 字段策略 |
|----|----------|
| Canonical | 解析后的 `canonical_custody_asset_code` / `identity_id` |
| 原始运营编号 | `imported_custody_asset_code`（建议新增）或审计列 |
| Operation | `source_asset_code`（资产编号(房源)权威列） |
| Persistence | `trust_asset_id` ← `primary_trust_asset_id` 或 upsert 回写 |

**禁止**用运营编号覆盖 Canonical 或发行 SSOT（Principle 2）。

发行模块：创建 Identity + 种子 Alias，**不走** Alias 反查。

## 七、Backfill Strategy

全部遵循 [Production Data Repair Standard](../engineering/production_data_repair_standard.md)。

| 阶段 | 动作 | 写库 |
|------|------|------|
| **B0** | 盘点 distinct custody；建立 **Identity** 候选集 | 否 |
| **B1** | 自 Issuance（SSOT 规则）写入 `asset_identities` + **Issuance Alias** 种子 | Repair Job |
| **B2** | 自 Monitor/Repayment 推断**候选** Alias（保守唯一归属） | 仅候选表 |
| **B3** | 人工确认候选 → 写入 `trust_asset_aliases` | 审计 |
| **B4** | 事实表 `identity_id` / `primary_trust_asset_id` 回填 | **Repair Job** |
| **B5** | 输出 **Identity Coverage Report**（覆盖率、未解析数） | 报告 |

**禁止** B2 自动 COMMIT 模糊匹配。参考案例：[`product3_repay_0612_custody`](../ops/fixes/product3_repay_0612_custody/README.md)（归属修正流程）。

## 八、Unresolved Queue

逻辑对象：**`AssetAliasUnresolved`**

| 状态 | 含义 |
|------|------|
| `open` | 待处理 |
| `confirmed` | 已确认映射并写入 Alias |
| `repaired` | 经 Repair 关闭 |
| `ignored` | 双人审批后统计排除 |

**输出：** `unresolved_custody_report_{product}_{date}.csv`

| 模式 | 行为 |
|------|------|
| 默认 | `needs_confirm` |
| 严格 | `reject` |

Precheck / B0 均写入或更新 Queue，避免静默错挂。

## 九、Layered Identity（Principle 8）

```
Business Layer     → Canonical Identity（asset_identities）
Operation Layer    → source_asset_code；运营托管编号（imported）
Persistence Layer  → trust_asset_id（trust_assets）
```

| 层 | 不得 |
|----|------|
| Canonical | 被 trust_asset_id 或运营 custody 替代 |
| source_asset_code | 作为 Canonical 或 Alias 键 |
| trust_asset_id | 作为开放 API 或 Workbench 业务主键 |

## 十七、Architecture Decision Records（ADR）

### ADR-001：Canonical 为什么不是 `trust_asset_id`？

`trust_asset_id` 属于 **Persistence Layer**，随 upsert 与历史数据质量变化，不具备业务终身稳定性。Domain 聚合必须以 **Canonical Identity** 为锚（Principle 8、11）。

### ADR-002：为什么不用运营期 `custody_asset_code` 作为 Canonical？

运营编号可随外部机构调整而变化（Principle 2）。Canonical 仅取**发行时**托管编号；运营编号通过 **Alias** 映射。

### ADR-003：为什么对外长期保留 Canonical 三元组？

保证开放 API、深链、BI 导出在多年演进后仍**稳定可对接**，不绑定内部 surrogate 生命周期（M3.0 §十三）。

### ADR-004：为什么需要 `identity_id`？

降低内部 Join、Alias 外键、投影缓存成本；**不改变** Canonical 业务语义。

### ADR-005：为什么 Alias 不自动猜测？

遵循 Principle 4 与 M3.0 §七：`ambiguous` / `unresolved` 必须 `needs_confirm` 或 `reject`。Product3 0612 类错挂证明静默解析代价极高。

### ADR-006：为什么 `business_asset_key` 不做主键？

M2 / M3.0 明确其仅用于**发行导入冲突分析**，发行表可多行同键。`identity_id` 在 Canonical 级 1:1，SSOT 规则在 §3.4 单独处理多行发行。
