## 二、Architecture Position

### 2.1 Workbench 是什么

```
┌─────────────────────────────────────────────────────────┐
│  Presentation（Web / Mobile / BI 壳层）                  │  ← M3.2 不定义视觉
├─────────────────────────────────────────────────────────┤
│  Asset Workbench Application Layer  ← 本文（M3.2）       │
│    Application Service + Projection 契约 + 导航          │
├─────────────────────────────────────────────────────────┤
│  Asset Domain（M3.0）                                    │
│    AssetIdentity（Root）· Alias · Lineage · 事实归属      │  ← M3.1 身份
├─────────────────────────────────────────────────────────┤
│  Persistence / 事实表（M2 字典）                         │
└─────────────────────────────────────────────────────────┘
```

| Workbench **是** | Workbench **不是** |
|------------------|-------------------|
| Asset Domain 的**第一层 Application** | 数据库视图或 ORM 聚合 |
| 围绕 **AssetIdentity** 的运营入口（M3.0 Principle 11） | Projection 存储本身（M3.3 实现） |
| Application Service 的**职责契约** | 独立业务事实源 |
| Web / API / AI 的**统一读模型入口** | Import / Alias 解析器（M3.1） |

### 2.2 与 Aggregate Root 的关系

所有 Workbench 读操作的路径：

```
输入（Canonical 三元组 或 identity_id）
    ↓
解析为 AssetIdentity（M3.1；Workbench 不猜测）
    ↓
Application Service 加载 Projection
    ↓
返回 Workbench DTO（Header / Snapshot / Timeline / …）
```

**禁止**绕过 `AssetIdentity` 用 `trust_asset_id` 或运营 `custody` 作为长期主入口（Dual Mode 过渡期见 §八）。

## 三、Workbench Goals

### 3.1 为什么需要 Asset Workbench

| 问题 | Workbench 目标 |
|------|----------------|
| 六套模块、多个标识符、多个入口 | **一个发行资产、一个运营视图** |
| 托管编号变更导致历史断裂 | 经 M3.1 Alias 归集后，Workbench **只展示 Canonical** |
| 逾期 / 风控 / 还款各做各的「详情页」 | **统一 Snapshot + Timeline**，减少重复聚合逻辑 |
| AI / BI 直接查表 | **Domain First**（M3.0 Principle 10），统一 Application Service |

### 3.2 为什么不能继续以 Overdue Workbench 为中心

| Overdue Workbench 现状 | 局限 |
|------------------------|------|
| 以最新监控快照 + `trust_asset_id` / `custody` 为入口 | 非发行锚点；编号变更后难串联全生命周期 |
| 侧重逾期跟进、核对 KPI | 不含完整 Issuance / 全量 Repayment 历史 / Trust 上下文 |
| 与 `/risk/workbench` 并列 | 两套队列、两套深链，AI 无法单一语义 |

Overdue **保留为业务队列**；**资产详情与运营动作**收敛至 Asset Workbench（`view=overdue` 切片）。

### 3.3 为什么所有模块统一进入 Asset Workbench

| 模块 | 在 Workbench 中的角色 |
|------|----------------------|
| Issuance | Snapshot 权威来源展示 |
| Monitor / Repayment | Snapshot + Timeline 事实投影 |
| Risk / Overdue | Snapshot 风险切片 + Operations |
| Import | Timeline 事件来源，不在 Workbench 内执行 |
| Repair | Timeline + Operations 审计 |

**一个 Aggregate、多种 View**，而非多个「工作台」。

## 四、Workbench Layout

六个主区域 + AI 辅区（M3.2 逻辑布局，非 UI 稿）。

```
┌──────────────────────────────────────────────────────────────┐
│  Header（Identity + 全局 as-of + 风险摘要）                    │
├────────────────────────────┬─────────────────────────────────┤
│  Snapshot（可折叠扇区）      │  Timeline（主历史流）             │
│  · Issuance                │                                 │
│  · Monitor                 │                                 │
│  · Repayment               │                                 │
│  · Risk                    │                                 │
│  · Trust                   │                                 │
│  · Operations 摘要         │                                 │
├────────────────────────────┴─────────────────────────────────┤
│  Operations（跟进 / 指派 / 提醒 / 手工动作）                  │
├──────────────────────────────────────────────────────────────┤
│  Attachments                                                 │
├──────────────────────────────────────────────────────────────┤
│  AI（摘要 / 问答 / 建议 — 只读，M3.5 增强）                   │
└──────────────────────────────────────────────────────────────┘
```

| 区域 | 职责 | 输入 | 输出 | 边界 |
|------|------|------|------|------|
| **Header** | Identity 展示、产品/发行日/Canonical、当前 as-of、内部状态 | `identity_id` 或三元组 | `AssetWorkbenchHeader` DTO | 只读 Identity；不编辑 Canonical |
| **Snapshot** | 各域当前切片投影 | Identity + `as_of_date?` | 各 `*Snapshot` DTO | **Projection only**（§五） |
| **Timeline** | 全生命周期事件流 | Identity + 分页游标 | `AssetEvent[]` | 可重建；禁止改事实（§六） |
| **Operations** | 可写运营动作入口 | Identity + 用户上下文 | 操作结果 + 新 Timeline 事件 | 写 Operations 事实表，不写 Snapshot |
| **Attachments** | 文件元数据与下载 | Identity | 附件列表 | 不存业务金额事实 |
| **AI** | 基于 Workbench 聚合结果的问答/摘要 | Identity + 已加载 Projection | 文本/结构化建议 | **禁止直读事实表**（§二十） |

## 五、Workbench Snapshot

### 5.1 Snapshot 原则（M3.0 Principle 7 / 5）

- Snapshot **不是事实数据**，是 **Projection**
- 来源于事实表 + M3.1 归集后的 Identity 范围
- 可缓存、可重建；事实表为唯一真相源
- Header 必须展示 **全局 as-of**（默认 Monitor 最新 `data_date`）

### 5.2 各 Snapshot 定义

| Snapshot | 时间语义 | 数据来源（事实） | 刷新方式 | 默认展示 |
|----------|----------|------------------|----------|----------|
| **Issuance** | `issue_date` 单次发行 | `trust_product_issuance_asset_records`（SSOT 行，M3.1 §3.4） | 发行导入后失效缓存 | 合同金额、转让价、迁移类型、地址等 |
| **Monitor** | 最新 `data_date`（可切换历史日期） | `trust_asset_monitor_records` + Identity 范围 | 监控导入 / 切换 as-of | 初受让、已还、剩余、逾期天数、最后回款日 |
| **Repayment** | 累计 + 可选 `repayment_date` 区间 | `trust_repayment_detail_records` | 还款导入 | 累计已还、期次分布、近期还款 |
| **Risk** | 绑定 Monitor as-of 或风险计算时刻 | `risk_alerts`、监控衍生指标、案件视图 | 风险重算 / 告警变更 | 等级、评分、开放告警数 |
| **Trust** | 产品级上下文（非单次快照） | `trust_products`、资产包关联、募集摘要 | 产品主数据变更 | 产品名、关联资产包、募集进度（只读） |
| **Operations** | 全量运营态 | `trust_overdue_followups` 等 | 跟进 CRUD | 未关闭跟进数、最近跟进摘要 |

### 5.3 Snapshot 扇区边界

- 各扇区 **独立懒加载**（§十一）
- 扇区之间 **禁止 N+1 跨表 Join 暴露给前端**；由 Application Service 分接口提供
- 跨扇区一致性（如 Monitor 已还 vs Repayment 合计）以 M3.0 核对口径为准，差异进 Timeline `RECONCILIATION` 类事件（可选 M3.3）

## 六、Timeline Design

### 6.1 定位（M3.0 §十 / Principle 9）

- 资产生命周期**唯一历史视图**（与 Operations 列表互补：Timeline = 事件，Operations = 可写任务）
- **Projection**：由事实表 + Alias 变更 + Repair 日志 **生成**，可全量重建
- **永远追加**；禁止覆盖、删除历史事件（归档除外）

### 6.2 Event Model

| 字段 | 类型 | 说明 |
|------|------|------|
| `event_id` | string | 全局唯一（建议 `{source_module}:{source_ref}` 或 UUID） |
| `identity_id` | int | 归属 Aggregate Root |
| `event_type` | enum | 见 §6.3 |
| `occurred_at` | datetime | 业务发生时间 |
| `recorded_at` | datetime | 系统记录时间 |
| `source_module` | string | `issuance` / `assetinfo` / `risk` / `ops` / `repair` |
| `source_ref` | string | 原表 PK、import batch、repair_name |
| `correlation_id` | string? | 同批次导入 / Repair |
| `payload_summary` | string | 展示用短描述 |
| `payload` | object? | 结构化扩展（M3.3 可选） |

### 6.3 事件类型（初版）

| event_type | 来源事实 |
|------------|----------|
| `ISSUANCE` | 发行记录创建 |
| `ALIAS_CHANGED` | `trust_asset_aliases` 新增/失效 |
| `MONITOR_IMPORTED` | 监控 Sheet 导入 |
| `REPAYMENT_IMPORTED` | 还款 Sheet 导入 |
| `RISK_CREATED` | 风险告警/案件 |
| `FOLLOWUP_CREATED` / `FOLLOWUP_UPDATED` | 逾期跟进 |
| `REPAIR` | `repair_log` / Repair Job |
| `IMPORT` | 通用导入批次（无细分时） |
| `MANUAL` | 人工 Operations 动作 |

### 6.4 排序、分页、Projection

| 项 | 规则 |
|----|------|
| 排序 | `occurred_at DESC`，tie-breaker `recorded_at DESC` |
| 分页 | Cursor（`occurred_at, event_id`）；默认 page size 50 |
| 生成 | M3.3：`build_timeline(identity_id)` 批处理或按需；M3.2 仅定义契约 |
| 重建 | 事实表不变前提下可全量重放；用于修 Projection bug |

## 七、Operations Design

### 7.1 定义

**Operations** = 人对资产的可写运营动作，**依附 AssetIdentity**，不修改 Canonical。

| 类型 | 事实 / 投影 | 说明 |
|------|-------------|------|
| **Follow-up** | 事实（`trust_overdue_followups`） | 跟进创建/更新/关闭 |
| **Comments** | 事实（未来 `asset_comments` 或扩展 followup） | 追加评论 |
| **Repair** | 事实触发 + Repair 审计 | 发起 Repair 工单，非 Workbench 内直接 UPDATE |
| **Manual Action** | 事实 + Timeline | 标记、信托反馈登记 |
| **Assignment** | 事实 | 指派处理人 |
| **Reminder** | 事实 / 调度 | 提醒任务 |
| **Audit** | 投影 + 审计表 | 展示谁何时做了什么 |

### 7.2 边界

- Workbench **编排** Operations，**不**替代各模块写库逻辑
- 写操作成功后 **追加 Timeline**（Principle 9）
- Operations **不**写入 Snapshot 缓存为事实

## 八、Navigation

### 8.1 统一入口 URL（Canonical 优先）

**主路径（M3.0 §十三）：**

```text
/asset-workbench
  ?trust_product_id=
  &issue_date=
  &canonical_custody_asset_code=
  &view=default|overdue|risk|issuance
  &as_of=YYYY-MM-DD          # 可选，Monitor Snapshot
```

**内部路径（Dual Mode，M3.1 §十）：**

```text
/asset-workbench?identity_id=123
/asset-workbench?trust_asset_id=456&trust_product_id=3   # 反查 Identity 后 302 到 Canonical
```

### 8.2 各入口收敛

| 来源 | 行为 |
|------|------|
| **Overdue 列表 / 工作台** | 重定向或嵌入 `view=overdue`；队列留 `/overdue`，详情进 Asset Workbench |
| **Risk Workbench** | `view=risk`；队列留 `/risk/workbench` 或合并队列 |
| **Issuance 列表** | 深链三元组（发行日 + custody） |
| **Monitor / Repayment 记录页** | 行级链接 → 解析 Identity → Workbench |
| **全局 Search** | 解析结果为 Identity 列表 → 进入 Workbench |
| **Open API（M3.4）** | 同 Application Service，非独立聚合 |
| **Deep Link** | 对外只文档化三元组；`identity_id` 仅内部 |

### 8.3 禁止

- 新增平行「资产详情页」绕过 Workbench
- 长期仅以 `custody_asset_code` 无 `trust_product_id` / `issue_date` 的深链

## 九、Application Service

Workbench **不**暴露 SQL 或表 Join；仅暴露下列**职责**（M3.4 映射为 REST，M3.5 映射为 AI Tool）。

| Service | 职责 | 输入 | 输出 |
|---------|------|------|------|
| `fetch_asset_identity()` | 加载 Aggregate Root 元数据 | 三元组或 `identity_id` | `AssetIdentityDTO`（含 Canonical、Alias 列表只读、SSOT 发行摘要） |
| `fetch_workbench_header()` | Header 区 | `identity_id`, `as_of?` | `AssetWorkbenchHeader` |
| `fetch_snapshot()` | 指定扇区 Snapshot | `identity_id`, `snapshot_type`, params | `IssuanceSnapshot` / `MonitorSnapshot` / … |
| `fetch_timeline()` | 事件分页 | `identity_id`, cursor | `TimelinePage` |
| `fetch_operations()` | 运营任务与权限 | `identity_id`, filters | `OperationsView` |
| `fetch_attachments()` | 附件元数据 | `identity_id` | `AttachmentList` |
| `fetch_ai_summary()` | 基于已加载 Projection 的摘要（M3.5） | `identity_id`, context | `AISummary`（只读） |

**聚合编排（可选内部，不暴露）：**

```text
fetch_asset_workbench() = identity + header + 默认 snapshot 元信息（不含全量 Timeline）
```

各扇区 **默认懒加载**，不鼓励单体巨型 DTO。

## 十、Projection Strategy

### 10.1 原则（M3.0 Principle 5 / 7 / 10）

| 允许 | 禁止 |
|------|------|
| 从事实表 **投影** 到 Snapshot / Timeline | Workbench Service **直接成为**新事实源 |
| Materialized View / 缓存表（M3.3） | 复制金额等业务事实到 Workbench 专用表且无溯源 |
| 按 Identity 失效缓存 | 跨 Identity 拼运营 custody Join |

### 10.2 刷新与重建

| 触发 | 动作 |
|------|------|
| 发行 / 监控 / 还款导入完成 | 失效相关 Snapshot 缓存；追加 Timeline 事件 |
| Alias 变更（M3.1） | 失效 Identity 相关全部 Projection |
| Repair 完成 | 失效 + `REPAIR` Timeline |
| 管理员 | `rebuild_projection(identity_id)`（M3.3） |

### 10.3 实现阶段

| 阶段 | 策略 |
|------|------|
| M3.2 首版 | 按需查询事实表 + 内存组装（只读 Workbench） |
| M3.3 | 可选 MV / `asset_timeline_events` 投影表 |
| M3.4+ | API 层缓存（ETag / short TTL） |

## 十一、Performance

| 原则 | 说明 |
|------|------|
| **Lazy Load** | Header 首屏；Snapshot 按扇区；Timeline 独立请求 |
| **Pagination** | Timeline 游标；Repayment Snapshot 内明细表格分页 |
| **Cache** | Snapshot 可 per-identity TTL；Import 后主动失效 |
| **禁止** | 一次加载全产品还款历史；全表 Scan 无 `identity_id` 过滤 |
| **as-of** | Monitor/Risk 默认最新 `data_date`；切换 as-of 重新拉取 |

## 十二、Security

| 域 | 要求 |
|----|------|
| **RBAC** | 读 Workbench：operator+；写 Operations：按角色 |
| **Audit** | 所有写操作记用户、时间、Identity；进 Timeline |

## 十六、Architecture Decision Records（ADR）

### ADR-WB-001：为什么 Workbench 不是事实层？

事实由各业务模块表持有；Workbench 聚合展示若写入独立「工作台表」将导致双源真相。遵循 M3.0 Principle 5 / 7。

### ADR-WB-002：为什么 Projection 独立于页面？

同一 Projection 服务 Web、API、BI、AI，避免四套 Join 逻辑漂移（Application Layer Baseline 定位）。

### ADR-WB-003：为什么引入 Application Service 层？

落实 Domain First（Principle 10）：Presentation 不感知表结构，只感知 Domain DTO。

### ADR-WB-004：为什么所有入口统一 Asset Workbench？

消除 Overdue/Risk 双工作台并行；AI 与开放 API 只需理解一种资产上下文。

### ADR-WB-005：为什么 Snapshot 分扇区懒加载？

性能与清晰 as-of 语义；避免首屏加载全量还款与全历史 Timeline。

### ADR-WB-006：为什么 Timeline 可重建仍要独立模型？

审计与 AI 需要稳定事件 schema；重建能力用于修 bug，不改变事实（Principle 9）。

### ADR-WB-007：为什么 Trust 进入 Snapshot 而非仅 Header？

运营人员需在资产上下文看到产品/资产包/募集，避免跳转多个系统。

## 十八、Asset Workbench UI Information Architecture

> **不是 UI 设计**（无颜色、组件、像素）；仅定义**信息架构**：区域、业务对象、数据来源。

### 18.1 页面信息块映射

| UI 区域 | 展示业务对象 | 数据来源 |
|---------|--------------|----------|
| Header | `AssetIdentity`、TrustProduct 名、`issue_date`、Canonical custody、风险徽章、as-of | `fetch_asset_identity` + `fetch_workbench_header` |
| Snapshot › Issuance | `IssuanceAsset`（SSOT 行） | `fetch_snapshot(issuance)` |
| Snapshot › Monitor | `MonitorSnapshot` | `fetch_snapshot(monitor)` |
| Snapshot › Repayment | 累计 + 明细表（分页） | `fetch_snapshot(repayment)` |
| Snapshot › Risk | `RiskAlert` 摘要、等级 | `fetch_snapshot(risk)` |
| Snapshot › Trust | `TrustProduct`、AssetPool 链接 | `fetch_snapshot(trust)` |
| Snapshot › Operations |  open follow-up 数、最近一条 | `fetch_snapshot(operations)` |
| Timeline | `AssetEvent[]` | `fetch_timeline` |
| Operations 面板 | Follow-up 表单、指派、评论 | `fetch_operations` + 写 API（M3.4） |
| Attachments | 文件列表 | `fetch_attachments` |
| AI 面板 | 摘要、问答 | `fetch_ai_summary` / M3.5 tools |

### 18.2 用户视图中禁止出现

- 裸 `trust_asset_id` 作为主标题（可作为技术 ID 折叠展示）
- 未解析的运营 custody 作为「资产主编号」
- 直接链接到 SQL 或表名

## 十九、Asset Workbench vs Existing Modules

| 现有模块 | 保留 | 收敛至 Workbench | 未来重定向 |
|----------|------|------------------|------------|
| **Overdue 列表** `/overdue` | 逾期队列表、筛选、标记 | 资产详情、跟进、核对 KPI 嵌入 `view=overdue` | 列表保留；详情链 → `/asset-workbench?...&view=overdue` |
| **Overdue Workbench** `/overdue/workbench` | Dual Mode 过渡期 | Snapshot/Timeline/Operations 能力 | 302 → Asset Workbench |
| **Risk Workbench** `/risk/workbench` | 风险队列（可选） | 风险 Snapshot、告警 Timeline | `view=risk` |
| **Issuance 导入/列表** | 导入流程、发行列表 | 单资产 Issuance Snapshot | 行点击 → 三元组深链 |
| **Monitor 记录页** | 表级查询、导入 | 单资产 Monitor Snapshot | 行点击 → Workbench |
| **Repayment 记录页** | 表级查询、导入 | Repayment Snapshot + Timeline | 行点击 → Workbench |
| **Import** | precheck / 导入执行 | 仅 Timeline 展示结果 | 不在 Workbench 内导入 |

**原则：** 列表与批量操作保留；**单资产运营上下文**统一 Workbench。

## 二十、AI Ready

### 20.1 为什么 Workbench 是 AI Agent 唯一入口

- AI 需要**同一资产**的 Issuance + Monitor + Repayment + Risk + 跟进 **语义完整上下文**
- M3.0 Principle 10：禁止 AI 直接读事实表
- Application Service 已做 Identity 归集与字段 Canonical 化

### 20.2 AI 调用约束

```
AI Agent
    ↓ 仅 Tool 调用
Workbench Application Service（§九）
    ↓
Projection / 事实（服务内部）
```

| 允许 | 禁止 |
|------|------|
| `fetch_asset_identity`、`fetch_snapshot`、`fetch_timeline`、`fetch_ai_summary` | 任意 SQL / ORM 直查 |
| 基于 `identity_id` 内部参数 | 绕过 Alias 用运营 custody 查全库 |
| 只读分析、生成跟进建议 | 通过 AI 执行 Repair 或改 Canonical |

### 20.3 M3.5 预留

- Tool 清单与 §九 一一对应
- Prompt 上下文 = Header + 相关 Snapshot 摘要 + Timeline 最近 N 条
- 引用字段遵循 M2 `field_dictionary`
